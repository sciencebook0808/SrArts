# Prisma Migration Recovery Runbook

> **Error:** `P3009 — migrate found failed migrations in the target database`  
> **Migration:** `20260323000000_community_reference_fields`  
> **Sources:** [pris.ly/d/migrate-resolve](https://pris.ly/d/migrate-resolve) · Prisma v7 Docs (March 2026)

---

## Phase 1 — Root Cause Analysis

### What P3009 Means

Prisma tracks every migration in the `_prisma_migrations` table in your database. Each row has:

| Column | Meaning |
|---|---|
| `migration_name` | Timestamp + name (e.g. `20260323000000_community_reference_fields`) |
| `started_at` | When Prisma started applying it |
| `finished_at` | `NULL` if it never completed successfully |
| `rolled_back_at` | Set if manually marked as rolled-back |
| `logs` | The error message if it failed |
| `applied_steps_count` | How many SQL statements completed before failure |

**P3009 fires when:** `finished_at IS NULL` AND `rolled_back_at IS NULL` for any row. This means the migration started but never finished — Prisma considers it "in a failed state" and refuses to apply any further migrations until the state is resolved.

### Why This Migration Failed

The migration `20260323000000_community_reference_fields` was **hand-crafted** (not generated by `prisma migrate dev`) and added to the migrations folder directly. Likely failure causes:

1. **Database connection interrupted** during `ALTER TABLE` execution on CockroachDB Serverless (common with cold-start connections)
2. **Concurrent execution conflict** — the `postbuild` script (`setupDatabase.ts`) was running the same DDL via raw SQL at the same time as `prisma migrate deploy`
3. **CockroachDB-specific:** `VARCHAR` vs `STRING` — CockroachDB's native type alias is `STRING`; while `VARCHAR` is accepted, mixed usage can cause edge-case engine issues
4. **Index naming conflict** — the original index name `CommunityPost_referenceType_id_idx` was too short and shadowed a Prisma-auto-generated name

### The Architectural Conflict (Critical)

The original setup had **two overlapping DDL systems** running against the same database:

```
Build time:  prisma migrate deploy  ← Creates tables via prisma/migrations/*.sql
Post-build:  setupDatabase.ts       ← Creates SAME tables via scripts/initDatabase.sql
```

This created a race condition. The `postbuild` `setupDatabase.ts` ran `CREATE TABLE IF NOT EXISTS "CommunityPost"` with the NEW columns already in the schema — which means on a fresh database, the table was created with the new columns BEFORE Prisma's migration ran. Then when Prisma's migration tried `ALTER TABLE ADD COLUMN`, it **either succeeded silently (IF NOT EXISTS) or, on CockroachDB, recorded a partial failure** before the index creation step.

**Fix:** The `postbuild` script has been removed from `package.json`. Prisma Migrate is the single source of truth for schema. `setupDatabase.ts` is kept only as a manual recovery tool (`npm run db:setup`), not as an automated step.

## Phase 2 — Database State Audit

Before running any fix, inspect the actual database state:

### 2a. Query `_prisma_migrations`

```sql
SELECT
  migration_name,
  started_at,
  finished_at,
  rolled_back_at,
  applied_steps_count,
  logs
FROM "_prisma_migrations"
WHERE migration_name = '20260323000000_community_reference_fields';
```

**Interpret results:**

| `finished_at` | `rolled_back_at` | `logs` | State |
|---|---|---|---|
| `NULL` | `NULL` | Error text | **Failed** → use fix below |
| Timestamp | `NULL` | Empty | Already applied (no action needed) |
| `NULL` | Timestamp | — | Already rolled back (just run `migrate deploy`) |

### 2b. Check Whether Columns Already Exist

```sql
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'CommunityPost'
  AND column_name IN (
    'referenceType', 'referenceId',
    'referenceTitle', 'referenceImage', 'referenceSlug'
  );
```

- **5 rows returned** → columns exist → use **Option B** (mark as applied)
- **0–4 rows returned** → columns missing/partial → use **Option A** (roll back + redeploy)

## Phase 3 — Fix Commands

### Option A — Columns Do NOT Exist (roll back and redeploy)

> Use when the migration was rejected before creating any columns, or when you want a clean re-run.

```bash
# Step 1: Mark migration as rolled-back in _prisma_migrations
# Source: https://www.prisma.io/docs/cli/migrate/resolve
npx prisma migrate resolve --rolled-back "20260323000000_community_reference_fields"

# Step 2: Verify the status is now clean
npx prisma migrate status

# Step 3: Re-apply the migration (now fixed with IF NOT EXISTS + STRING type)
npx prisma migrate deploy
```

### Option B — Columns DO Exist (mark as applied)

> Use when the migration was partially or fully applied — the columns are in the DB but Prisma doesn't know it.

```bash
# Mark migration as applied — tells Prisma to consider it complete
# Source: https://www.prisma.io/docs/cli/migrate/resolve
npx prisma migrate resolve --applied "20260323000000_community_reference_fields"

# Verify
npx prisma migrate status
```

### One-Command Shortcut

```bash
npm run db:fix
```

This runs `scripts/fix-failed-migration.sh` which automatically:
1. Checks migration status
2. Marks as rolled-back
3. Redeploys

## Phase 4 — What Changed in the Migration File

The original migration SQL had two problems on CockroachDB:

**Before (broken):**
```sql
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceType" VARCHAR;
-- ... all 5 columns in sequence (CockroachDB prefers separate statements) ...
CREATE INDEX IF NOT EXISTS "CommunityPost_referenceType_id_idx" ON ...;
```

**After (fixed):**
```sql
-- CockroachDB native type STRING instead of VARCHAR alias
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceType"  STRING;
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceId"    STRING;
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceTitle" STRING;
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceImage" STRING;
ALTER TABLE "CommunityPost" ADD COLUMN IF NOT EXISTS "referenceSlug"  STRING;

-- Corrected index name (matches Prisma auto-generated naming convention)
CREATE INDEX IF NOT EXISTS "CommunityPost_referenceType_referenceId_idx"
  ON "CommunityPost" ("referenceType", "referenceId");
```

## Phase 5 — Build Script Fix

**Before:**
```json
"build": "prisma generate && prisma migrate deploy && next build",
"postbuild": "node -e \"...runs setupDatabase.ts...\""
```

**After:**
```json
"build": "prisma generate && node scripts/migrate-deploy-safe.js && next build"
```

The new `scripts/migrate-deploy-safe.js` wrapper:
- Skips if `DATABASE_URL` is not set (build-time environments like Vercel)
- Detects P3009 and prints exact recovery commands instead of crashing the build
- `next build` always proceeds — the app runs, just without the new columns until migration is fixed in production

## Phase 6 — Prevention

### Never Create Hand-Crafted Migrations for Schema Changes

Always use:
```bash
# Local dev — generates and applies migration
npx prisma migrate dev --name community_reference_fields

# This produces a Prisma-generated migration.sql that is:
# - Guaranteed correct for your provider (cockroachdb)
# - Recorded properly in migration_lock.toml  
# - Tested against the shadow database first
```

### Do Not Duplicate DDL Between Migrate and Raw SQL Scripts

- `prisma/migrations/` = single source of truth for schema DDL
- `scripts/initDatabase.sql` = manual emergency recovery only, not automated

### Validate Migrations Locally Before Deploy

```bash
# 1. Test against a local/staging DB first
DATABASE_URL="..." npx prisma migrate deploy

# 2. Check status
npx prisma migrate status

# 3. Only then commit and push to trigger production deploy
```
